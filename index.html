Images - Modern Setup

Combining modern and efficient CSS and HTML habits looks like this:

html

<div class="container">
  <!-- Attributes here define the SHAPE for the browser -->
   <img src="beach-small.jpg" width="300" height="200"
    ="Sunny beach">
</div>

css

.container img {
  width: 100%; /* Fill the div width */
  height: auto; /* Or a fixed height like 200px */
  object-fit: cover; /* Fix any ratio mismatch */
}

Summary

HTML attributes (width and height) prevent the page from jumping.

A file size appropriate for the use of the image makes the page load fast. So adapt the size of the files you set ready for upload, according to their intended use.

CSS (object-fit) makes the image look pretty in its box.

Javascipt - For adding interactivity to your pages

JavaScript is the programming language that adds interactivity and dynamic behavior to an HTML document, transforming a static page into a responsive user experience.

What is JavaScript used for ?

While HTML defines the structure and CSS handles the styling, JavaScript provides the "brain" by allowing the page to:

  • Respond to user actions like clicks, scrolls, and typing.
  • Update content in real-time without reloading the entire page.
  • Manipulate elements directly through the Document Object Model (DOM).

Linking

A modern way of linking an external JavaScript file is to place the <script> element in the <head> section and use the defer or async attribute to prevent blocking HTML parsing.

  Use defer for scripts that depend on the DOM or when execution order matters.
  Use async otherwise (for independent scripts).

But in modern application code, the preferred approach is to use type="module".

Module scripts are deferred by default, execute after HTML parsing, preserve execution order, and support import/export, making them cleaner and better structured.

async remains useful for independent third-party scripts (analytics, ads), and defer is still relevant for classic (non-module) scripts that rely on the DOM.

Example using 'type=module', to be added at the end of the <head> section:

  <script type="module" src="app.js" ></script>

Note: ES modules may not always load, for example when opening a local HTML file directly in the browser, because browsers block module scripts for security reasons.

Styling Links: A Guide to CSS Pseudo-Classes

In CSS, a pseudo-class is a keyword added to a selector that specifies a special state of the element being selected.

Links (the <a> tag) go through several states as users interact with them. These states are targeted using pseudo-classes to provide visual feedback and improve accessibility for keyboard and screen reader users.

The five primary link states are:

  • :link (Unvisited): The default state for a link that has not been clicked yet. By default, browsers typically display these as blue and underlined.
  • :visited: A link the user has previously opened. To protect user privacy, browsers limit the CSS properties you can change here (mostly just colors).
  • :hover: Triggered when a user moves their mouse pointer over the link without clicking it. This provides a visual cue that the element is interactive.
  • :focus: Occurs when a link is selected via keyboard navigation (usually the Tab key). This is critical for accessibility to show keyboard users where they are on the page.
  • :active: The momentary state while a link is actually being clicked or activated. In default browser styling, these often turn red.

The "LVFHA" Ordering Rule

Because later CSS rules can override earlier ones, you must define these in a specific order to ensure they work correctly. A common mnemonic for this is "LoVe Fears HAte":

  1. Link
  2. Visited
  3. Focus
  4. Hover
  5. Active

If you define :visited after :hover, for example, the hover effect might never appear for links the user has already seen.

CSS Selectors

Descendant combinator

li em { color: rebeccapurple; }

This selector focuses on em elements that are found inside li elements

Next-sibling combinator

h1 + p { font-size: 200%; }

Selects p elements that are found just after h1 elements

Selectors based on state

a:link { color: pink; }

a:visited { color: green; }

a:hover { text-decoration: none; }

The links will appear pink at start but green when visited.
They will appear underlined but not when hovered.

Combined selectors

h1 + p .special {
  color: navy;
  border-radius: 3px;
  border: 1px solid navy;
  }

The style is applied to elements with special class found inside p elements that immediately follow an h1 heading.

Note: Not to be confused with h1 + p.special {} which targets paragraphs marked by the class special and which directly follow a h1 element. Just one space making all the difference!

The box model

Any html element behave like a box.

It is important to understand this behaviour, especially later when dealing with more complex layouts like CSS grid or flexbox.

Block and inline boxes

The boxes in which most HTML elements reside can be either block boxes or inline boxes

Block Boxes follow all box model rules. They stack vertically and respect every dimension or margin you set.

Inline Boxes only respect horizontal properties (left/right margin and padding). They effectively ignore width, height, and any vertical margin.

The "Middle Ground": If you want an element to stay in a line of text but still respect width, height, and vertical padding/margin, you can use display: inline-block.

Inner and outer display types

In modern CSS, the display property is technically a multi-keyword property that defines two distinct behaviors at once: how a box acts toward its neighbors (outer) and how it organizes its children (inner).

The outer display type determines how the element participates in the page's "Normal Flow" (how it sits next to other boxes).

The inner display type defines the formatting context created inside the box for its direct children.

Changing the display property determines the way an element behaves along its neighborhood as well as how its children behave in it as a container.

About cascading and inheritance

In CSS, the "cascade" refers to the specific algorithm the browser uses to resolve conflicts when multiple styles apply to the same HTML element. It determines which rule "wins" and which property values are ultimately applied.

The browser follows a predictable hierarchy, often compared to a waterfall where styles "flow" down through several levels of priority.

The Core Cascading Factors

When two or more rules conflict, the browser checks these criteria in order:

  • Importance & Origin: The browser first considers where the style comes from. For example, a developer's stylesheet (Author origin) overrides the browser's default styles (User-Agent origin). Rules marked with !important jump to the top of this priority list.
  • Specificity: If the origins are equal, the more targeted selector wins. For instance, a rule using an ID selector (#header) is more specific and will override a class selector (.header) or a tag selector (h1).
  • Source Order: If importance and specificity are identical, the last rule the browser encounters in the code "wins". Newer styles effectively overwrite previous ones.
  • Cascade Layers: Modern CSS allows developers to group styles into explicit layers using @layer. Styles in a layer defined later have higher priority regardless of their internal specificity.

Distinction from Inheritance

While often discussed together, cascading is different from inheritance.

  • Cascading is about resolving conflicts between different rules targeting the same element.
  • Inheritance is the process by which an element automatically receives certain properties (like color or font-family) from its parent element.

Note: In fact, instead of seeing CSS cascading as a conflict solving system (among all those possible property values, which one shall I choose?) one could also see it as an algorithm that always determine the most appropriate value, the most targeted one. Instead of a conflict, it becomes a general to specific approach. Seeing it so makes it probably easier to understand how CSS works, all along.

The alternative CSS box model

Historical context

The original CSS box model (content-box) was defined in the early days of the web, when CSS was mainly used for text styling rather than layout.

In this model:

  • width and height apply only to the content
  • padding and border are added on top

As web design evolved toward more complex layouts, this behavior proved counterintuitive.

To address this, CSS introduced an alternative:

👉 box-sizing: border-box

This model aligns better with how developers naturally think about element size.

MDN recommended syntax

html {
  box-sizing: border-box;
}

*,
*::before,
*::after {
  box-sizing: inherit;
}

What the alternative model does

With:

box-sizing: border-box;

The declared width/height becomes the total size of the element, including:

  • content
  • padding
  • border

So:

  • Final width = declared width
  • Content shrinks to fit inside

Result:

The size you set is the size you get.

Why use the alternative model

  • Predictability: avoids unexpected overflow due to padding/borders
  • Simplicity: easier mental model for layout
  • Consistency: widely used in modern development and frameworks

👉 It matches intuitive thinking:

“I define the final size of the element.”

Why use the MDN pattern

While slightly more complex, instead of forcing border-box everywhere directly, the MDN approach

  • sets it once on html and
  • makes all elements inherit.

Advantages:

  • Single source of truth
      → behavior is defined at the root
  • Flexibility
      → easy to switch a section back to content-box

    .component {
      box-sizing: content-box;
    }

  • Better maintainability
      → avoids heavy overrides in complex projects
  • Consistency for pseudo-elements
      → ::before and ::after behave the same way

Key takeaway

One-line memory

Use border-box so the size you set is the size you get.

About the nature of float

A floated element is removed from the normal layout flow. While subsequent block-level elements occupy their normal positions as if the float weren't there, their internal line boxes (the invisible horizontal strips that wrap each row of content) are shortened to prevent their inline content—like text or images—from colliding with the float.

Its a bit like the float element was glued upon the other elements and in addition acted as a magnet upon certain rendered contents, like text ?

That is a fantastic way to visualize it. You’ve hit on the "magic" of floats.

To use your analogy:

  • The Glue: The floated element is "glued" on top of the normal layout. Because it is pulled out of the standard flow, the paragraph's block box doesn't even see it—it just slides right under the float as if it weren't there.
  • The Magnet: The float then acts like a magnetic field that only affects inline content (text and line boxes). As the line boxes try to render, they "feel" the magnet and are pushed away, shortening themselves to avoid a collision.

This is exactly why you can have a paragraph with a background color that sits behind an image, while the text itself neatly wraps around it. The "container" ignores the glue, but the "content" reacts to the magnet.

The only way to make the paragraph box "stop" being ignored by the glue is to create a Block Formatting Context (BFC), which essentially tells the paragraph: "Don't slide under that float; respect its boundaries as a whole box."

Positioning

The position property defines the coordinate system of an element, determining how it sits in the page layout or if it "breaks out" of it entirely. It allows you to move elements to specific locations using top, bottom, left, and right relative to their parent, the viewport, or their own original spot.
